## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Form group gaps (`.form-group gap: 4px`), inline badge padding |
| sm | 8px | Button footer gaps, element spacing within cards |
| md | 16px | Card padding (canonical: `.card` uses `18px 20px` — normalize to `16px`), section divider margin, modal body gap |
| lg | 24px | Top nav padding, modal padding, `acct-content` top padding |
| xl | 32px | Account header padding sides (`28px 32px 24px`), tab content padding sides |
| 2xl | 48px | Not currently used — reserve for future page-level spacing |
| 3xl | 64px | Not currently used — reserve for future page-level spacing |

Exceptions:
- Touch targets for sidebar items: minimum 44px tall (7px top + 7px bottom padding + 13px font = ~27px actual; increase to `padding: 10px 14px` to reach 44px for accessibility)
- Sidebar section header: `padding: 0 12px 8px` — keep as-is (this is a label row, not a tap target)
- Card border-radius: 8px for `.exec-card`, 10px for `.card` and `.modal` — retain existing radii, do not normalize to a single value

Source: index.html lines 83–370, CONTEXT.md D-17

| Role | Value | CSS Token | Usage |
|------|-------|-----------|-------|
| Dominant (60%) | `#080C16` | `--dark` | Page background, overlays (`rgba(8,12,22,0.85)`) |
| Secondary (30%) | `#141B2D` / `#1A2236` | `--surface` / `--card` | Sidebar background (`--surface`), cards and panels (`--card`), AI panel |
| Accent (10%) | `#3B82F6` | `--accent` | Reserved list below |
| Destructive | `#EF4444` | `--red` | Destructive button borders/text, form error text, remove account hover |

Accent (`--accent` #3B82F6 and `--accent2` #60A5FA) reserved for:
- Active sidebar item border-left and text color
- Active tab underline and text color
- Focused form input border
- Primary `.btn-primary` background
- Tag backgrounds and borders (`.tag` uses `rgba(59,130,246,0.1)`)
- Signal icon backgrounds
- Inline edit input focus ring (new in this phase — see Inline Edit below)
- Category dropdown focus state (new in this phase)

Status colors (do not use for non-status purposes):
- `--green` (#22C55E): New exec badge, approve button, positive KPI deltas
- `--gold` (#F59E0B): Warning states, neutral-positive signals
- `--red` (#EF4444): Destructive actions only — delete, reject, error text

## Component Contracts

### Shared Card Class (`.card`)

Canonical definition (normalize any per-tab deviations to this):
```
background: var(--card)
border: 1px solid var(--border)
border-radius: 10px
padding: 16px
```
Note: existing `.card` has `padding: 18px 20px` — normalize to `16px` on all sides for consistency across tabs. Card label (`.card-label`) stays at `font-size: 11px; font-weight: 500; color: var(--muted)`.

### Section Header (`.section-header`) — new shared class

Used for in-tab heading rows that label a data group:
```
font-size: 11px
font-weight: 500
text-transform: uppercase
letter-spacing: 0.08em
color: var(--muted)
margin-bottom: 8px
```

### Inline Form (`.inline-form`) — existing per-tab pattern to normalize

Form fields rendered directly in tab panes (contacts, activity, strategy) must share:
```
.form-group gap: 4px
.form-label font-size: 11px; font-weight: 500; color: var(--muted)
.form-input padding: 8px 12px; font-size: 13px
.form-input:focus border-color: var(--accent)
```

### Action Button (`.acct-action-btn`) — existing, retain as-is

No changes. Already consistent across panels.

## New Interaction Contracts

### Category Dropdown in Edit Modal (UIPOL-02)

- Location: existing account edit modal, below the sector text field (replace sector free-text with a `<select>` dropdown)
- Populated from: distinct `sector` values returned by a `GET /api/categories` endpoint
- Includes: "New category..." option at the bottom of the list as the last `<option>`
- When "New category..." selected: a text input appears inline below the `<select>` with placeholder "Type new category name"
- Input appearance: same `.form-input` styling as all other modal inputs
- Save behavior: calls existing `PUT /api/accounts/:id` with updated `sector` value; sidebar re-renders via `renderSidebar()` immediately
- Error state: if save fails, show `.form-error` under the dropdown with text "Failed to move account. Please try again."
- Visual cue for the new inline text input: fade-in at 150ms (`opacity 0→1, translateY 4px→0`)

### Inline Category Rename (UIPOL-03)

- Trigger: double-click on a `.sidebar-section` header text node (not the chevron)
- Visual transition to edit state: section header text node is replaced in-place with a `<input type="text">` using `.sidebar-rename-input` class
- `.sidebar-rename-input` appearance:
  - `background: transparent`
  - `border: none`
  - `border-bottom: 1px solid var(--accent)`
  - `color: var(--text)`
  - `font-size: 10px; font-weight: 500; text-transform: uppercase; letter-spacing: 0.1em`
  - `font-family: var(--font)`
  - `outline: none`
  - `width: 100%`
- Confirm: Enter key or blur (clicking away)
- Cancel: Escape key — restores original text, no API call
- On confirm: calls `PUT /api/categories/rename` with `{ oldName, newName }`; re-renders sidebar on success
- On API error: show a transient toast message "Rename failed. Please try again." — 3 second duration (see Toast below)
- Bulk rename is atomic server-side — no partial state shown to user

### Toast Notification (new micro-component)

Used for category rename error feedback. No confirmation required for success — sidebar re-render is sufficient confirmation.

- Position: bottom-right, `position: fixed; bottom: 24px; right: 24px; z-index: 2000`
- Appearance: `background: var(--card); border: 1px solid var(--border2); border-radius: 8px; padding: 12px 16px; font-size: 13px; color: var(--text)`
- Error variant: left border `3px solid var(--red)`
- Success variant: left border `3px solid var(--green)` (available for future use)
- Animation: fade-in 150ms, auto-dismiss after 3000ms with fade-out 150ms
- No dismiss button required (auto-dismiss only)

## Visual Consistency Audit Targets (UIPOL-01)

The following specific inconsistencies must be resolved during the audit pass. This is prescriptive — executor does not need to discover these:

| Tab | Inconsistency | Required Fix |
|-----|--------------|--------------|
| All tabs | `.card` padding varies (`18px 20px` vs `16px` vs `12px 14px`) | Normalize `.card` padding to `16px` universally |
| Contacts tab | Section headers use ad-hoc inline styles | Apply `.section-header` class |
| Activity tab | Entry spacing uses margin-bottom inconsistently | Use `8px` gap on the entry container |
| Strategy tab | Inline form inputs may differ from `.form-input` spec | Audit and apply `.form-input` class |
| Briefing tab | `.briefing-section-header` font-size is 15px | Retain 15px — this is a document heading role, not a card label |
| Overview tab | `.stat-val` at 22px is the only 22px usage | Normalize to 20px (display role) if it reads well at that size; if not, retain 22px and document as exception |
| All tabs | `font-size: 12px` appears in `.bar-row` and `.topnav-count` | Normalize to 11px (label role) |
