**A. Add in-memory sliding window rate limiter (D-11 through D-16):**

Near the top of server.js (after the existing const declarations), add:

```javascript
const rateLimitMap = new Map();
const RATE_LIMIT = parseInt(process.env.RATE_LIMIT_PER_MINUTE) || 10;

function checkRateLimit(req, res) {
  var cookies = getCookies(req);
  var sessionId = cookies['gd_auth'] || 'anonymous';
  var now = Date.now();
  var windowStart = now - 60000;
  var timestamps = (rateLimitMap.get(sessionId) || []).filter(function(t) { return t > windowStart; });
  if (timestamps.length >= RATE_LIMIT) {
    logger.log('warn', req.url, 'RATE_LIMITED', 'Rate limit exceeded for session', null);
    res.writeHead(429, { 'Content-Type': 'application/json' });
    res.end(JSON.stringify({
      error: 'Too many requests. Please wait before trying again.',
      code: 'RATE_LIMITED'
    }));
    return false;
  }
  timestamps.push(now);
  rateLimitMap.set(sessionId, timestamps);
  return true;
}
```

Per D-11: No npm dependencies. Per D-13: Default 10/min, configurable via RATE_LIMIT_PER_MINUTE. Per D-14: Returns 429 with JSON. Per D-16: In-memory only.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> server AI endpoints | User requests cross into server-side AI processing with cost implications |
| server -> Anthropic API | Outbound requests to external API with potential timeout/failure |
| server error responses -> client | Error messages cross back to untrusted client |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-06 | Denial of Service | AI endpoints | mitigate | Rate limiter (checkRateLimit) enforces 10 req/min per session via sliding window. Returns 429 before any Anthropic API call is made. Prevents token cost runaway. |
| T-07-07 | Information Disclosure | Error responses | mitigate | All error responses use structured JSON with generic user-friendly messages (D-10). Raw error details (e.message, stack traces, API keys) logged server-side only via logger.log(), never included in HTTP response body. Validated by grep for 'detail.*e.message' absence. |
| T-07-08 | Denial of Service | Anthropic API timeout | mitigate | 55s timeout on all https.request calls. Timeout fires proxy.destroy() which closes the socket. headersSent guard prevents double response. Without timeout, a hung Anthropic connection would hold the Node.js request indefinitely. |
| T-07-09 | Spoofing | Rate limiter session ID | accept | Session ID is the gd_auth cookie value. In a shared-password app, all users share one rate limit bucket. This is acceptable -- the rate limit protects against rapid automated requests, not per-user fairness. If multi-user auth is added later, rate limiting automatically becomes per-user. |
| T-07-10 | Tampering | rateLimitMap in-memory | accept | In-memory Map resets on server restart (D-16). An attacker could restart the server to reset limits -- but server restart requires Railway access (admin only). |
</threat_model>
