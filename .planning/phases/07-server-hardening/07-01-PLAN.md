<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| server.js -> filesystem | Log writes and backup copies to Railway Volume |
| env vars -> backup/logger config | BACKUP_DIR, LOG_PATH, BACKUP_INTERVAL_HOURS from environment |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Tampering | logger.js log file | accept | Log file is append-only on Railway Volume; no user-facing write path. Railway Volume access is admin-only via dashboard/shell. |
| T-07-02 | Denial of Service | logger.js rotateIfNeeded | mitigate | Cap log file at 10MB with single rotation file (errors.log.1). Maximum disk usage: ~20MB. Prevents unbounded growth. |
| T-07-03 | Information Disclosure | logger.js entries | mitigate | Log entries include endpoint, error_type, message, account_id only. Never log request bodies, API keys, or user credentials. The log() function constructs entries from explicit parameters -- no automatic req/res dumping. |
| T-07-04 | Denial of Service | backup.js disk usage | mitigate | KEEP_COUNT=5 with pruning. Max backup storage: 5 x DB size. For a <10MB DB, that is <50MB. Prune runs after every backup. |
| T-07-05 | Elevation of Privilege | backup.js path traversal | accept | BACKUP_DIR comes from env var (admin-controlled), not user input. Path constructed server-side with path.join(). No user-controlled path segments. |
</threat_model>
