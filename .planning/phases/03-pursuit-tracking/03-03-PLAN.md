Add a POST /api/accounts/:id/debrief endpoint in server.js. Place it AFTER the activity routes (from Plan 01) and BEFORE the /api/claude proxy endpoint. This endpoint sends the debrief text to the Anthropic API and returns structured proposals -- it MUST NOT write anything to the database (per D-15, PURS-03).

Add the route matcher:
```javascript
const debriefMatch = parsed.pathname.match(/^\/api\/accounts\/([a-z0-9-]+)\/debrief$/);
```

Handler for POST /api/accounts/:id/debrief:
1. Read body with readBody()
2. Parse JSON, extract `text` field. Validate text is a non-empty string (400 if missing).
3. Validate account exists via db.getAccount() (404 if not found).
4. Fetch recent activity for context: `db.getActivity(accountId, 10)` -- last 10 entries.
5. Fetch contacts for context: `db.getContacts(accountId)` -- to help AI identify known vs new contacts.
6. Build the system prompt:

```javascript
var systemPrompt = GD_CONTEXT +
  '\n\nACCOUNT: ' + account.name +
  '\nSECTOR: ' + account.sector +
  '\nHQ: ' + account.hq +
  '\nREVENUE: ' + account.revenue +
  '\n\nACCOUNT INTELLIGENCE:\n' + account.context;

if (recentActivity.length > 0) {
  systemPrompt += '\n\nRECENT ACTIVITY:\n' + recentActivity.map(function(a) {
    return a.created_at + ' [' + a.type + '] ' + a.summary;
  }).join('\n');
}

if (contacts.length > 0) {
  systemPrompt += '\n\nKNOWN CONTACTS:\n' + contacts.map(function(c) {
    return c.name + ' - ' + c.title + ' (' + c.influence + ')';
  }).join('\n');
}
```

7. Build the user message requesting structured JSON output:

```javascript
var userMessage = 'Extract structured activity log entries from the following meeting debrief. ' +
  'Return ONLY valid JSON with no additional text, no markdown code fences. Use this exact format:\n' +
  '{"activities": [{"date": "YYYY-MM-DD", "type": "meeting|call|email|note|other", ' +
  '"participants": "Name (Title), Name (Title)", "summary": "What happened and key takeaways", ' +
  '"action_items": "Next steps"}], ' +
  '"contact_updates": [{"action": "new|update", "name": "Full Name", ' +
  '"title": "Job Title", "influence": "Champion|Evaluator|Blocker", ' +
  '"changes": "what changed (for updates only)"}]}\n\n' +
  'If no contact updates are needed, return an empty contact_updates array.\n' +
  'Embed action items naturally in the summary text.\n\n' +
  'DEBRIEF:\n' + debrief_text;
```

8. Send to Anthropic API using the same https.request pattern as /api/contacts/:id/generate. Use model `claude-sonnet-4-20250514` and `max_tokens: 4000` (larger than usual because the response needs to contain multiple structured entries).

9. Parse the AI response text. Try JSON.parse() first. If that fails, try extracting JSON from between code fence markers (the AI sometimes wraps in ```json ... ```). If still unparseable, return 422 with `{ error: 'Could not parse AI response', raw: responseText }`.

10. On success, return 200 with the parsed JSON proposals. The response shape should be:
```json
{
  "activities": [...],
  "contact_updates": [...]
}
```

CRITICAL: This endpoint does NOT call db.addActivity() or any other write function. It only reads (getAccount, getActivity, getContacts) and returns the AI's proposals to the client.

**rejectEntry(accountId, index)** function (per D-13):
1. Remove the proposal card from DOM
2. Remove from pendingProposals[accountId].activities[index]

**approveAll(accountId)** function (per D-12):

4. Remove proposal on success

**rejectContactUpdate(accountId, index)** function:
1. Remove from pendingProposals and DOM

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User debrief text to AI | Untrusted text sent as part of AI prompt -- potential prompt injection |
| AI response to client | AI-generated JSON rendered in browser -- XSS risk |
| Client review panel to API | Approved entries sent from client to activity/contacts endpoints |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-08 | Tampering | Prompt injection via debrief text | accept | AI output is always reviewed by human before DB write (PURS-03 review gate). The human gate is the mitigation -- even if AI returns malicious content, user must approve it. |
| T-03-09 | Tampering | XSS via AI-extracted fields in review panel | mitigate | escapeHtml() applied to all AI-generated fields (summary, participants, date, contact names) before innerHTML injection |
| T-03-10 | Tampering | Modified proposal data on approve | mitigate | Server-side validation on POST /api/accounts/:id/activity validates type, summary, source. Same validation as manual entries. |
| T-03-11 | Denial of Service | Large debrief text overwhelming AI context | mitigate | Textarea limited to 10,000 characters client-side. readBody() 1MB limit server-side. Recent activity capped at 10 entries in system prompt. |
| T-03-12 | Repudiation | AI-generated entries indistinguishable from manual | mitigate | source field stored as 'ai_debrief' and ai_raw preserves original debrief text. UI shows "AI" badge on debrief-sourced entries. |
</threat_model>
